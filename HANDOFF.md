# HANDOFF — Статусы заказов через Telegram

**Дата:** 2026-05-31 · **Статус:** внедрено и проверено на проде (ritualb2b.ru).
**Спека:** `docs/superpowers/specs/2026-05-31-telegram-order-statuses-design.md`

## Что делает
Оператор управляет статусом заказа inline-кнопками прямо из Telegram. `orders.status` —
единый источник правды; его читают админка и личный кабинет клиента.

Поток (4 статуса, как на splithub):
```
Новый  →  [✅ Подтвердить]  →  [📦 Выполнен]
              │                    
              └─ [❌ Отменить] на любом шаге → заказ закрыт (кнопки исчезают)
```
После смены статуса в группу прилетает ответ `SH-000XX → <Статус>`, а клиент сразу
видит новый бейдж статуса в личном кабинете.

## Файлы
- **`send.php`** — после создания заказа залогиненного клиента вешает кнопки
  `[✅ Подтвердить][❌ Отменить]` на уведомление в Telegram (`editMessageReplyMarkup`).
  Гостевые заказы (без числового id) кнопок не получают.
- **`api/tg_poll.php`** — poll-обработчик. По cron раз в минуту дёргает `getUpdates`,
  обрабатывает нажатия, меняет `orders.status`, перерисовывает клавиатуру и шлёт ответ
  в группу. offset хранится в `app_settings` и сохраняется ВСЕГДА (защита от спама).
  Идемпотентен: повторное нажатие того же статуса = только всплывашка, без дубля.
- **`config.php`** — секреты (`BOT_TOKEN`, `CHAT_ID`, `EMAIL_TO`, `WEBHOOK_SECRET`).
  **НЕ в git** (`.gitignore`). Живёт только на сервере в корне сайта.

Админка (`api/admin.php`) и ЛК (`index.html` + `api/auth.php?action=history`) статусы
показывали и до этого — там ничего менять не пришлось.

## Эксплуатация
Ничего настраивать не нужно: при новом заказе залогиненного клиента в группе появляется
заявка с кнопками. Жмёте кнопку → в течение ≤1 минуты статус меняется и виден клиенту.
Промежуточные статусы «В обработке»/«Отгружен» доступны только вручную в админке.

## Деплой и обслуживание
- Код деплоится через git: `git push origin main`, затем на сервере `git pull origin main`
  под пользователем сайта.
- **`config.php` не деплоится через git** (gitignored) — он должен лежать на сервере вручную.
  При переустановке/переезде сайта пересоздать его с актуальными секретами.
- На сервере включён **cron раз в минуту**, который вызывает `api/tg_poll.php?secret=…`.
- Лог обработчика: `api/tg_poll.log` (пишутся только ошибки; тоже gitignored).

## Важные решения (грабли)
- **Polling, не webhook.** РФ-хостинг не пропускает входящие от Telegram — webhook не работает,
  поэтому опрос через cron. Перед первым запуском разово снят webhook (`deleteWebhook`).
- **Эндпоинт защищён** `?secret=<WEBHOOK_SECRET>` (без секрета → 403) и жёсткой сверкой `chat_id`.
- **Номер заказа** в ответах: `SH-` + id, дополненный нулями до 5 знаков (как в ЛК).

## Как проверялось
E2E на проде: создавались тестовые заказы залогиненного клиента, нажимались кнопки в группе,
подтверждалось через клиентский API (`auth.php?action=history`), что статус менялся
`new→confirmed→completed` и `new→cancelled`, ответные сообщения приходили, идемпотентность
работала (`errors=0`, offset растёт). Тестовые данные после проверки удалены.
